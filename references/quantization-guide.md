# 量化级别选择指南

GGUF 格式模型提供多种量化级别，需要在文件大小和模型质量之间取舍。

---

## 自动推荐逻辑

**默认量化级别：Q4_K_M**（体积与质量的最佳平衡）

根据用户计算机硬件自动推荐更合适的级别：

### 第一步：检测硬件

```bash
# === Linux ===
free -h
nvidia-smi --query-gpu=memory.total --format=csv,noheader

# === Windows (Git Bash) ===
# GPU 显存
nvidia-smi --query-gpu=memory.total --format=csv,noheader

# 内存（Git Bash 中 wmic 输出 GBK 编码会乱码，用 systeminfo 代替）
systeminfo 2>/dev/null | grep -i "物理内存\|Physical Memory" || \
  cat /proc/meminfo 2>/dev/null | grep -i "MemTotal\|MemAvailable"
```

如果检测到 GPU，以 **GPU 显存** 为主要依据（模型优先加载到显存）。没有独立 GPU 时以**系统内存**为依据。

### 第二步：根据硬件推荐

| 可用显存/内存 | 推荐量化 | 说明 |
|-------------|---------|------|
| < 4 GB | ⚠️ 不建议运行 7B+ 模型 | 考虑更小的模型（如 1.5B、3B） |
| 4-6 GB | Q3_K_M | 勉强可用，质量损失较明显 |
| 6-8 GB | **Q4_K_M**（默认） | 体积与质量的最佳平衡 |
| 8-12 GB | Q5_K_S | 明显提升，推荐 |
| 12-16 GB | Q6_K | 接近无损 |
| 16-24 GB | Q8_0 | 几乎无质量损失 |
| 24+ GB | F16（可选） | 原始精度，文件最大 |

> 需要为系统和其他程序预留约 2-4 GB 内存。上表已考虑预留量。

### 第三步：告知用户

向用户报告推荐结果和理由，例如：
- "检测到 8 GB 显存，推荐 Q4_K_M（~3.5 GB），留有余量"
- "检测到 16 GB 内存（无独立 GPU），推荐 Q5_K_S（~4.5 GB）"

如果用户有其他偏好，以用户选择为准。

## 量化级别对照表

| 量化级别 | 相对大小 | 质量 | 备注 |
|---------|---------|------|------|
| **Q3_K_M** | ~30% | 较低 | 损失明显，仅在内存极度受限时使用 |
| **Q4_K_M** | ~35% | 良好 | **默认推荐**，体积与质量的最佳平衡点 |
| **Q4_K_S** | ~33% | 良好 | 牺牲少量质量换取更小体积 |
| **Q5_K_S** | ~40% | 很好 | 比 Q4 明显提升 |
| **Q5_K_M** | ~42% | 很好 | Q5 系列中质量最优 |
| **Q6_K** | ~48% | 优秀 | 接近无损 |
| **Q8_0** | ~55% | 极好 | 几乎无质量损失 |
| **F16** | ~100% | 完美 | 原始精度，文件最大 |

## 按用途选择

在硬件允许的前提下，可根据用途微调：

| 用途 | 推荐量化 | 原因 |
|------|---------|------|
| 日常对话、写作 | Q4_K_M | 足够好，节省空间 |
| 代码生成 | Q5_K_M 以上 | 代码对精度更敏感 |
| 翻译 | Q4_K_M | 翻译任务对量化不敏感 |
| 数学推理 | Q6_K 以上 | 推理能力对精度要求高 |
| 长文本处理 | Q5_K_S 以上 | 长上下文对量化损失更敏感 |

## 命名规律

GGUF 文件名通常包含量化信息，帮助快速识别：

```
Qwen3-8B-Q4_K_M.gguf
│       │  └── 量化级别
│       └── 参数量（8B = 80 亿参数）
└── 模型名
```

常见后缀含义：
- **K** = K-quant（一种量化算法）
- **M** = Medium（中等精度）
- **S** = Small（较小体积）
- **L** = Large（较高质量）
- **0** = 基础量化（如 Q8_0）
